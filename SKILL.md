# Any2MD

Use this skill when Hermes needs one stable document-processing entrypoint instead of hardcoding direct calls to many external tools.

Goal:
- route documents to the best available backend
- preserve business structure before wiki ingestion
- normalize all backends into one stable schema
- keep install, upgrade, and doctor flows bundled with the tool
- default to local, no-auth backends where possible
- support one-step online installation for publishable GitHub usage

## When To Use

Use this skill for:
- PPT / PPTX with process diagrams, architecture maps, cards, lanes, or timelines
- legacy PPT / DOC / XLS that should be converted into modern formats before routing
- PDF documents that came from PPT and lost structure during plain-text conversion
- complex Word / Excel exports where hierarchy or table structure matters
- pre-processing before `wiki-ingest` creates concepts, entities, and links

Do not use this skill for:
- quick bulk conversion where full-text search is enough
- simple prose-heavy Word files that `markitdown` already converts cleanly

## Outputs

For each input file, generate:
- `*.structured.json`: canonical machine-readable structure
- `*.structured.md`: debug-oriented normalized markdown with machine detail
- `*.ext.md`: final human-readable markdown using the original source filename plus `.md`
- `*.mermaid.md`: optional Mermaid views for process / architecture / timeline pages
- `*.baseline.md`: optional baseline markdown from `markitdown`, if available
- `manifest.json`: batch manifest with chosen backend, attempts, and outputs

## Execution

Primary entrypoint:

```bash
./any2md "/abs/path/to/file-or-folder" "/abs/path/to/output"
```

Health check:

```bash
./any2md doctor
```

Read these references only when needed:
- extraction routing and backend rules: `references/backend-selection.md`
- output contract for downstream wiki extraction: `references/output-schema.md`
- install and adapter strategy: `README.md`

Install bootstrap:

```bash
bash install.sh
```

Optional:

```bash
bash install.sh --with-backends core
bash install.sh --with-backends full
bash install.sh --from-archive /path/to/downloads --with-backends recommended
```

## Workflow

1. Run `doctor` to detect which backends are currently available.
2. Detect the document family from extension.
3. Route to the configured backend chain from `config/defaults.toml`.
4. Try the first available specialist backend.
5. Fall back to local adapters if specialist tools are missing or fail.
6. Inspect `manifest.json` and `*.structured.json` if extraction quality matters.
7. Use `*.ext.md` as the final readable file.
8. Feed `*.ext.md` or `*.structured.json` into downstream wiki ingestion.

Default backend order:
- `pdf -> docling -> local-pdf -> markitdown`
- `docx -> docling -> markitdown`
- `pptx -> mineru -> native-pptx -> markitdown`
- `xlsx -> mineru -> markitdown`
- `md/default -> markitdown`

Legacy office handling:
- `ppt -> convert to pptx -> pptx backend chain`
- `doc -> convert to docx -> docx backend chain`
- `xls -> convert to xlsx -> xlsx backend chain`

## Quality Rules

- Prefer local specialist tools first, but never let Hermes depend on one vendor-specific output shape.
- Keep local fallbacks for at least PPTX and PDF.
- Keep raw evidence separate from structured outputs.
- Preserve source coordinates when the active backend exposes them.
- Make backend order configurable; do not hardcode it in Hermes prompts.
- For GitHub-published usage, online installation must be the default path; local archives are only a fallback.
- Treat `*.structured.md` as a debug artifact, not the final user-facing result.

## Recommended Pairing With markitdown

Use both together:
- `markitdown` gives broad compatibility and quick baseline text.
- `Any2MD` restores hierarchy, spatial grouping, and graph-like structure.

If `markitdown` is installed, this skill can invoke it automatically as a fallback backend.
